Arquitetura Core + módulos.

Não é monolito acoplado nem microserviço improvisado. É uma plataforma modular com contratos fortes, governança e evolução gradual — resumo do documento de arquitetura do LUMEN.

“O LUMEN não deve voltar ao modelo de monolito acoplado. A arquitetura-alvo é Core Platform + módulos independentes, com contratos fortes, governança obrigatória e evolução gradual.”

A composição

Quatro peças, papéis claros.

A plataforma é composta por um Core que entrega o transversal, módulos de negócio autônomos e dois serviços externos de IA e workflow.

Dentro do Core

Três artefatos, ciclos de vida diferentes.

O Core não é uma coisa só. São três artefatos independentes — não confundir: a API, o shell e o kit de componentes.

Core API lumen-core (backend)

Serviço que roda como container. A fonte da verdade de acesso.

Responsável por autenticação, autorização, RBAC, multiempresa, registry de módulos, auditoria, roteamento de notificações, configurações globais e service accounts. Expõe endpoints HTTP consumidos pelos módulos e pelo Core Frontend.

Core Frontend lumen-core (frontend)

O shell da plataforma.

Login, layout, sidebar, topbar, menu dinâmico e roteamento entre módulos. Não contém componentes de domínio nem lógica de negócio. Consome o @lumen/ui como dependência de pacote.

@lumen/ui pacote de componentes

O kit reutilizável — consumido pelo Core e por todos os módulos.

Biblioteca de componentes visuais, publicada como pacote NPM interno. Sem dependência do Core API em runtime.

A fronteira

O Core é plataforma, não negócio.

O que é comum a todos vive no Core. Regra de domínio nunca entra nele — isso é dos módulos.

✓ O Core fornece

  • ✓ Identidade, autenticação e autorização
  • ✓ Multiempresa e matriz de permissões
  • ✓ Registro de módulos e menu dinâmico
  • ✓ Auditoria, rastreabilidade e correlation_id
  • ✓ Configurações globais e contratos de integração
  • ✓ Governança técnica e base para alta disponibilidade

✕ O Core NÃO contém

  • ✕ Regra financeira, de contrato ou de licitação
  • ✕ Lógica de SLA, de chamado ou de RH
  • ✕ Cálculo de margem ou de folha
  • ✕ Lógica de IA transacional
  • ✕ Acesso direto ao banco interno de um módulo
  • ✕ Componentes visuais de domínio (esses são do @lumen/ui)

O princípio central

Modularidade real.

Cada módulo é independente — de verdade. Não depende de tabela, migration, service, model ou regra de outro módulo.

· Código-fonte· Repositório Git· Docker· API· Banco de dados· Migrations· Permissões· Eventos· Configurações· Deploy· Testes· Observabilidade

Como se comunicam

Contratos fortes entre módulos.

A conversa entre Core e módulos passa por contrato explícito — nunca por atalho no banco.

✓ Permitido

  • API HTTP interna versionada
  • Eventos, webhooks internos e filas
  • Contratos OpenAPI
  • Schemas de payload versionados

✕ Proibido

  • JOIN direto entre bancos de módulos
  • Leitura direta de tabela de outro módulo
  • Import de código interno de outro módulo
  • Endpoint não documentado / contrato verbal

Evolução gradual

Começar simples, crescer com segurança.

A primeira fase roda com um stack enxuto — banco separado por módulo numa mesma instância PostgreSQL — e evolui sob checklist.
